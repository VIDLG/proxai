# 流式行为

当客户端向 ProxAI 请求流式响应时，ProxAI 会保留上游 Server-Sent Events (SSE) 字节流，同时观察协议事件。从使用者视角看，三件事最重要：流最终会结束、错误以可读形态到达、卡住的 tool call 不会让客户端永远等待。

<ProxaiCallout type="note" title="你不需要学 ProxAI 内部实现">
  本页描述的是可观察的流式行为。内部流式架构请看 [流式内部实现](/zh/streaming-internals)。
</ProxaiCallout>

## 健康的流是什么样

每个 provider 协议都有自己的 terminal event。只有收到预期的 terminal event，ProxAI 才认为这条流完整。这实现了 [行为契约 C19](/zh/reference/behavior-contracts#流式)。

## 卡住的工具调用

如果上游开始流式发送 function/tool arguments 但一直没有完成，ProxAI 可以用协议形态的错误关闭流，而不是让客户端一直挂起。它由以下配置控制：

```toml
[tool_calls]
timeout_secs = 120
```

这是语义超时，不是整条请求的总时长上限。它只在工具调用参数流已经开始后才生效。这实现了 [行为契约 C20](/zh/reference/behavior-contracts#流式)。
