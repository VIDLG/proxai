# 配置说明

Runtime configuration centers on `config.toml`. The generated `config.example.toml` is the local example/reference; `config.toml` itself is intentionally git-ignored. These behaviors implement [Behavior Contract C1–C4](/zh/reference/behavior-contracts#配置与启动).

<ProxaiCallout type="tip" title="配置是长期行为的来源">
  长期运行行为应放在 `config.toml`。CLI flags 更适合临时覆盖，例如本地测试或一次性的 route 调整。完整示例参考见 [配置示例](/zh/reference/configuration-example)。
</ProxaiCallout>

## 应用目录

运行时文件生成在用户应用目录：

| 平台 | 路径 |
|---|---|
| Windows | `%USERPROFILE%\\.proxai\\` |
| Linux/macOS | `~/.proxai/` |

重要条目：

| 路径 | 用途 |
|---|---|
| `config.toml` | 运行时配置，刻意不进入 git |
| `config.example.toml` | 生成的本地示例/参考 |
| `logs/` | 运行日志输出 |
| `captures/` | 可选的分阶段 capture artifact |

## `[server]`

控制本地代理监听地址和请求接入限制。

| 字段 | 含义 |
|---|---|
| `host` | 本地绑定 host |
| `port` | 本地代理端口 |
| `max_request_body_bytes` | ProxAI 接受的最大入站请求体大小 |
| `max_concurrent_requests` | 本地代理同时处理的最大请求数 |

## `[mcp]`

控制本地 MCP 控制/API 监听器。

| 字段 | 含义 |
|---|---|
| `host` | 本地 MCP 绑定 host |
| `port` | 本地 MCP 端口 |

默认 MCP endpoint：

```text
http://127.0.0.1:18081/mcp
```

## `[routing.default_provider_names]`

按入站请求协议声明 fallback provider。只有没有显式 route 命中时才使用默认 provider。

当前 key：

- `openai_responses`
- `openai_chat_completions`
- `anthropic_messages`

Route 语义和示例见 [路由与 Provider](/zh/routing-and-providers)。

## `[[routing.routes]]`

Route 是对入站请求的过滤规则。

| 字段 | 含义 |
|---|---|
| `name` | 可选的稳定标识，用于 CLI override |
| `request_protocol` | 可选入站协议保护条件 |
| `match_kind` | 可选 `exact`、`glob`、`regex` 或 `auto` |
| `model_pattern` | 逻辑模型选择器 |
| `provider` | route 命中时选择的 provider 名称 |
| `upstream_model` | 可选上游模型映射 |

## `[providers.<name>]`

每个 provider 描述 ProxAI 如何连接一个具体上游。

| 字段 | 含义 |
|---|---|
| `protocol` | 出站 wire protocol：`openai_responses`、`openai_chat_completions` 或 `anthropic_messages` |
| `base_url` | 上游 provider base URL |
| `api_key` | ProxAI 使用的 provider API key |
| `compatibility` | 可选兼容模式，例如 `strict` 或 `anthropic_compatible` |
| `read_idle_timeout_secs` | 等待上游响应 bytes 时的 idle-read timeout |

<ProxaiCallout type="caution" title="不要提交密钥">
  `api_key` 只应存在于本地 `config.toml`，不要放进被跟踪的示例、日志、文档或 fixture。
</ProxaiCallout>

## `[tool_calls]`

| 字段 | 含义 |
|---|---|
| `timeout_secs` | 不完整 streamed tool-call arguments 的语义超时 |

它不同于 `read_idle_timeout_secs`：后者观察原始上游 bytes，`tool_calls.timeout_secs` 观察工具调用语义是否完成。

## `[capture]`

Capture phase 开关：

| 字段 | 捕获内容 |
|---|---|
| `inbound_request_enabled` | ProxAI 收到的客户端请求 |
| `provider_request_enabled` | 经过路由、翻译、模型改写和 provider 序列化后的请求 |
| `upstream_response_enabled` | 上游响应 headers 和原始 bytes |
| `outbound_response_enabled` | 最终返回给客户端的响应 |

Capture 路径固定在应用目录下。隐私规则和 CLI 命令见 [观测与诊断](/zh/observability)。

## `[logging]` 与 `[logging.duration_thresholds]`

| 字段 | 含义 |
|---|---|
| `output_format` | `human` 适合人工排查，`json` 适合机器消费 |
| `use_color` | 启用/禁用 human 日志颜色 |
| `warn_ms` / `error_ms` | human 日志耗时着色阈值 |

## `[error_responses]`

| 值 | 含义 |
|---|---|
| `text` | 简洁、适合 Zed 阅读的错误 body |
| `json` | 面向非 Zed 客户端的 OpenAI 风格 JSON 错误 body |

上游状态处理和保留 header 规则见 [常见排障](/zh/troubleshooting)。
