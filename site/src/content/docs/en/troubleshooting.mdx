# Troubleshooting

This page collects the most common ProxAI symptoms and points to the right next step. It does not re-explain protocol internals or configuration fields; each card links to the page that already owns that explanation.

<ProxaiCallout type="tip" title="Capture before guessing">
  Most symptoms are easiest to diagnose with one sanitized capture. Use [Observability](/en/observability) to enable `inbound_request`, `provider_request`, `upstream_response`, or `outbound_response` for a single request, then disable it again.
</ProxaiCallout>

## Request shape and routing

  items={[
    {
      symptom: 'Client sees 400 invalid_request_error',

    },
    {

      causes: ['Route order or model pattern matched an unexpected route.', '`request_protocol` guard mismatched the inbound endpoint.'],

    },
    {

    },
  ]}
/>

## Provider and upstream

<TroubleshootingGrid
  items={[
    {
      symptom: 'Client sees 401 / 403 upstream',
      icon: 'key',
      causes: ['Provider `api_key` is missing, wrong, or expired.', 'Upstream rejects the ProxAI-supplied auth header.'],
      next: ['Check `[providers.<name>].api_key` in local config.', 'For OpenAI providers, ProxAI sends Authorization: Bearer.', 'For Anthropic providers, ProxAI sends x-api-key.'],
    },
    {
      symptom: 'Client sees 429 upstream',
      icon: 'timer',
      causes: ['Upstream rate limit or quota exhausted.', 'ProxAI forwarded `Retry-After` if the upstream provided it.'],
      next: ['Inspect preserved `Retry-After` and rate-limit headers.', 'Check provider quota.', 'Lower concurrency in `[server].max_concurrent_requests` if useful.'],
    },
    {
      symptom: 'Client sees 502 / 503 upstream_error',
      icon: 'terminal',
      causes: ['Provider returned a non-2xx status.', 'Upstream error body was empty, non-JSON, or an unknown shape.'],
      next: ['Check provider `base_url`.', 'Enable `upstream_response` capture for one request.', 'Confirm the upstream endpoint is reachable.'],
    },
  ]}
/>
