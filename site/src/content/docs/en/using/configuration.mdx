# Configuration

Runtime configuration centers on `config.toml`. Use this page for common configuration tasks; use [Configuration Reference](/en/reference/configuration) for the complete generated `config.example.toml` and [Defaults and Limits](/en/reference/defaults-and-limits) for exact default values.

<ProxaiCallout type="tip" title="Config is the source of truth">
  Keep long-lived runtime behavior in `config.toml`. CLI flags are temporary overrides for local testing or one-off route changes.
</ProxaiCallout>

## Minimal local config

A minimal useful setup needs a local listener, default provider mapping, and at least one provider.

```toml
[server]
host = "127.0.0.1"
port = 18080

[routing.default_provider_names]
openai_responses = "openai"
openai_chat_completions = "openai"
anthropic_messages = "anthropic"

[providers.openai]
protocol = "openai_responses"
base_url = "https://api.openai.com"
api_key = "..."

[providers.anthropic]
protocol = "anthropic_messages"
base_url = "https://api.anthropic.com"
api_key = "..."
```

<ProxaiCallout type="caution" title="Never commit secrets">
  `api_key` values belong in local `config.toml`, not in tracked examples, logs, docs, captures, or fixtures.
</ProxaiCallout>

## Add a provider

Each provider describes one concrete upstream. This page keeps only the quick shape; use [Provider Setup](/en/using/provider-setup) for provider naming, auth headers, defaults, and route examples.

```toml
[providers.openai]
protocol = "openai_responses"
base_url = "https://api.openai.com"
api_key = "..."
```

Provider names are labels. Provider `protocol` controls outbound wire behavior. See [Glossary](/en/reference/glossary) for the distinction between provider name and provider protocol.

## Set default providers

`[routing.default_provider_names]` declares fallback providers per inbound request protocol. Defaults are used only when no explicit route matches.

```toml
[routing.default_provider_names]
openai_responses = "openai"
openai_chat_completions = "openai_chat"
anthropic_messages = "anthropic"
```

Supported keys are the protocol values from [Protocols](/en/reference/protocols):

- `openai_responses`
- `openai_chat_completions`
- `anthropic_messages`

## Add an explicit route

Use an explicit route when a model pattern needs a specific provider, optional inbound protocol guard, or upstream model rewrite.

```toml
[[routing.routes]]
name = "m3_chat"
request_protocol = "openai_chat_completions"
match_kind = "exact"
model_pattern = "MiniMax-M3-preview"
provider = "minimax"
upstream_model = "MiniMax-M3"
```

## Tune streaming timeouts

There are two different timeout concepts:

<ReferenceTable
  columns={['Setting', 'Layer', 'Purpose']}
  rows={[
    ['[tool_calls].timeout_secs', 'Protocol semantics', 'Timeout for incomplete streamed tool-call arguments'],
    ['[providers.<name>].read_idle_timeout_secs', 'Transport carrier', 'Idle-read timeout while waiting for upstream response bytes'],
  ]}
/>

Keep them separate: `read_idle_timeout_secs` watches raw upstream bytes, while `tool_calls.timeout_secs` watches semantic tool-call completion.

## Enable captures

Capture phase switches live under `[capture]`:

```toml
[capture]
inbound_request_enabled = false
provider_request_enabled = false
upstream_response_enabled = false
outbound_response_enabled = false
```

## Configure logging

`[logging]` controls output format and color behavior. `[logging.duration_thresholds]` controls human-log duration coloring.

<ReferenceTable
  columns={['Field', 'Meaning']}
  rows={[
    ['output_format', '`human` for interactive debugging or `json` for machine consumption'],
    ['use_color', 'Enable or disable colored human logs'],
    ['warn_ms / error_ms', 'Human log duration coloring thresholds'],
  ]}
/>

Logs should stay compact and should not include request bodies, API keys, Authorization headers, or private prompts.

## Choose error response format

`[error_responses]` selects how client-facing errors are rendered.

<ReferenceTable
  columns={['Value', 'Meaning']}
  rows={[
    ['text', 'Concise Zed-readable error bodies; default'],
    ['json', 'OpenAI-style JSON error bodies for non-Zed clients'],
  ]}
/>

```toml
[error_responses]
format = "text" # or "json"
```

For upstream status handling and preserved headers, see [Error Responses](/en/reference/error-responses).
