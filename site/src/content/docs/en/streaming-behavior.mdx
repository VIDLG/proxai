# Streaming Behavior

When a client asks ProxAI for a streamed response, ProxAI preserves the upstream Server-Sent Events (SSE) byte stream while observing protocol events. From a user perspective, three things matter: the stream eventually ends, errors arrive in a readable shape, and stalled tool calls do not hang forever.

<ProxaiCallout type="note" title="You do not need to learn ProxAI internals">
  This page describes observable streaming behavior. For internal streaming architecture, see [Streaming Internals](/en/streaming-internals).
</ProxaiCallout>

## What a healthy stream looks like

Each provider protocol has its own terminal event. ProxAI considers a stream complete only when the expected terminal event arrives. This implements [Behavior Contract C19](/en/reference/behavior-contracts#streaming).

## Stalled tool calls

If a provider starts streaming function/tool arguments but never finishes them, ProxAI can close the stream with a protocol-shaped error instead of hanging the client forever. This is controlled by:

```toml
[tool_calls]
timeout_secs = 120
```

The timeout is semantic, not a total request timeout. It only applies after tool-call argument streaming has started. This implements [Behavior Contract C20](/en/reference/behavior-contracts#streaming).
